# 企业级年报查询系统 — RAG 知识库建设注意事项

> 阶段目标：先完成知识库与检索能力，暂不搭建前端网站。  
> 源文件格式：PDF、DOC/DOCX。  
> 核心要求：文字、表格、图片/图表等多模态信息均可被有效切片与检索。

---

## 一、年报文档本身的特殊性

### 1.1 结构与版式复杂

| 特征 | 对 RAG 的影响 |
|------|----------------|
| 固定章节（董事会报告、财务摘要、附注等） | 需保留章节层级元数据，便于按章节过滤检索 |
| 跨页表格、合并单元格 | 简单按页/按段切片会打碎表格语义 |
| 图表与正文强关联（「见图 3」「如下表所示」） | 图文块需建立引用关系，避免检索时上下文断裂 |
| 页眉页脚、页码、水印 | 解析时需清洗，防止污染向量与关键词索引 |
| 中英文混排、专业术语密集 | 分词与 Embedding 模型需适配中文财经语境 |

### 1.2 信息类型多样

年报不仅是「段落文字」，至少包含：

- **叙述性文本**：经营分析、风险披露、ESG 叙述
- **结构化表格**：资产负债表、利润表、现金流量表及附注明细
- **图表**：营收趋势、占比饼图、组织架构图
- **列表与编号条款**：会计政策、重大事项编号
- **扫描件/图片型 PDF**：部分历史年报或盖章页需 OCR

**注意**：不能假设「全文 OCR 成纯文本再切片」就足够；表格行列关系、图表数值趋势会在此流程中严重丢失。

### 1.3 时效性与版本

- 同一公司存在多年年报，检索时常带 **年份、报告期** 约束
- 修订版、更正公告、摘要版与全文版可能并存
- 必须在元数据中明确：`公司代码`、` fiscal_year`、`document_type`、`version`、`source_file_hash`

## 二、文档解析阶段注意事项

### 2.1 PDF 与 DOC 需区别对待

| 格式 | 常见坑 | 建议 |
|------|--------|------|
| **文本型 PDF** | 阅读顺序错乱（多栏、浮动图） | 使用带版面分析的工具（非单纯 `pdfplumber.extract_text`） |
| **扫描型 PDF** | 无文本层 | OCR + 版面检测；评估 OCR 准确率对财务数字的影响 |
| **DOC/DOCX** | 复杂样式、嵌入对象 | 优先转统一中间格式（如 Markdown/HTML + 结构化 JSON），避免多套解析逻辑 |

**原则**：DOC/DOCX 建议先转为 PDF 或统一 HTML/MD 流水线，降低下游切片逻辑分叉。

### 2.2 表格处理（高优先级）

- **禁止**将表格当作普通段落按字数切块
- 每个逻辑表应作为 **独立 Chunk**，并同时保存：
  - 人类可读的 Markdown 表
  - 机器可读的结构化表示（JSON：headers + rows + 单位/币种）
- 跨页表需 **合并后再切片**，表题（caption）与表体不可分离
- 附注中的「续表」需识别并关联主表

### 2.3 图片与图表处理

- **装饰性图片**（Logo、分隔线）：可忽略或仅存元数据
- **信息型图片**（趋势图、示意图、照片说明）：必须处理
  - 方案 A：多模态 Embedding（图+文联合向量）
  - 方案 B：VLM 生成 **结构化图注**（图表类型、轴含义、关键数据点）再入文本索引
  - 方案 C：图表旁提取到的表格数据单独结构化存储
- 图片 Chunk 应保留：`page_no`、`figure_id`、`caption`、`related_section`

### 2.4 切片（Chunking）原则

| 原则 | 说明 |
|------|------|
| **语义完整** | 一个 Chunk 应能独立回答某一类问题，避免半句话、半张表 |
| **层级保留** | 记录 `section_path`（如 `第三章 > 财务摘要 > 营业收入`） |
| **父子块** | 小块用于精准检索，大块（Parent）用于生成时补全上下文 |
| **类型标签** | `chunk_type`: `text` / `table` / `figure` / `list` |
| **重叠策略** | 叙述文本可适度 overlap；表格与图表 **不要** 重叠切片 |
| **Token 上限** | 受 Embedding 模型限制；表格过大时按「行组」或「科目组」子切，但保持表头 |

### 2.5 元数据设计（检索过滤的生命线）

建议每条 Chunk 至少包含：

```yaml
company_id: "600519.SH"
company_name: "贵州茅台"
fiscal_year: 2023
report_type: "annual"          # annual / interim / prospectus
language: "zh"
section: "第三节 管理层讨论与分析"
section_path: ["第三节", "管理层讨论与分析", "营业收入分析"]
page_start: 42
page_end: 43
chunk_type: "table"
chunk_id: "uuid"
parent_chunk_id: "uuid"      # 可选
source_file: "2023年报.pdf"
source_hash: "sha256:..."
parse_pipeline_version: "v1.2.0"
```

企业级场景下，**元数据过滤 + 向量检索** 的组合往往比纯向量检索更稳定。

## 三、检索与 Agent 阶段注意事项

### 3.1 纯向量检索的局限

年报问答典型问题：

- 「2022 与 2023 年净利润对比」→ 需跨 Chunk、跨表聚合
- 「研发费用占营收比例趋势」→ 需读表 + 计算
- 「风险因素第三条是什么」→ 需精确章节定位

因此需要：

- **混合检索**：BM25（关键词）+ 稠密向量 + 可选稀疏向量（SPLADE 等）
- **重排序（Reranker）**：粗召回 Top-K 后精排
- **查询改写**：口语化问题 → 财经术语（Agent 或规则）

### 3.2 Agent 不是万能，要设边界

- Agent 适合：**多步检索、工具调用、计算、对比、引用溯源**
- Agent 不适合：替代高质量的文档解析与索引；垃圾进 = 垃圾出
- 必须实现 **引用溯源（Citation）**：回答需标注来源 Chunk、页码、章节
- 财务类回答建议增加 **置信度与免责**：数字类结果尽量来自结构化表 Chunk，而非模型臆造

### 3.3 幻觉与数字准确性

- 对 **金额、百分比、同比** 等：优先从结构化表 Chunk 读取，Agent 仅做编排与解释
- 可引入 **校验工具**：检索结果中的数字与源表 JSON 交叉验证
- 生成温度应偏低；关键路径可强制 JSON 输出

### 3.4 权限与合规（企业级必项）

- 年报虽多为公开信息，但企业内控常要求：**按公司/部门/角色** 控制可见范围
- 操作审计：谁问了什么、返回了哪些文档片段
- 数据驻留：Embedding 与原文是否允许出域/上公有云
- 个人信息：年报中董监高薪酬等是否需脱敏

## 四、数据质量与运营注意事项

### 4.1 解析质量评估

在上线前对样本集做人工标注评估：

| 维度 | 指标示例 |
|------|----------|
| 文本提取 | 段落完整率、乱序率 |
| 表格 | 单元格对齐准确率、表头识别率 |
| 图表 | 图注完整性、关键数值召回 |
| 切片 | 语义完整度（人工打分） |
| 检索 | Recall@K、MRR、答案归因正确率 |

### 4.2 增量更新与全量重建

- 新一年年报入库：增量解析 + 索引
- 解析管线升级：需支持 **按 `parse_pipeline_version` 重跑**
- 保留原始文件与中间产物（Parsed JSON），便于排错与重索引

### 4.3 成本控制

- VLM 解析全量图表成本高：可先规则筛选「含坐标轴/图例」的信息图再调用
- Embedding 批量离线计算；在线只算 Query 向量
- 大表 Chunk 过多时：摘要索引 + 明细表分级存储

### 4.4 可观测性

- 记录每条问答的：检索 Query、召回 Chunk ID、Rerank 分数、LLM Prompt 哈希
- 解析失败队列：OCR 低置信页、表格破碎页人工复核

## 五、阶段边界（当前不做的事）

为聚焦 RAG 知识库，**本阶段明确排除**：

- 前端网站与账号体系（仅预留 API 契约）
- 复杂对话产品化（多轮记忆、订阅推送）
- 实时行情或公告流接入（可二期与 RAG 并联）

**本阶段交付物建议**：

1. 文档解析与多模态切片管线（可批处理）
2. 向量库 + 关键词索引 + 元数据存储
3. 检索 API（含混合检索与 Rerank）
4. Agent 编排 PoC（命令行或 Notebook 验证即可）
5. 评估数据集与质量报告

## 六、风险清单（简表）

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 扫描 PDF OCR 数字错误 | 高 | 双引擎 OCR + 置信度阈值 + 人工抽检 |
| 表格结构丢失 | 高 | 专用表格提取 + 结构化双写 |
| 图表无法检索 | 中 | VLM 图注 + 多模态向量 |
| 跨年对比问答错误 | 高 | 元数据过滤 + Agent 多步检索 + 计算工具 |
| 向量模型财经术语弱 | 中 | 领域微调 Embedding 或 Reranker |
| 索引膨胀 | 中 | 分层索引、冷热分离、按年分区 |

---

## 七、成功标准（建议）

知识库阶段可验收的标准：

1. 给定 100 份 PDF/DOC 年报，自动入库成功率 ≥ 95%（允许少量扫描件进人工队列）
2. 核心财务表（三大报表）结构化还原准确率 ≥ 98%（样本人工核验）
3. 测试集 200 个问题，检索阶段相关 Chunk 进入 Top-5 的比例 ≥ 85%
4. Agent PoC 对「带引用的事实性问题」回答准确率 ≥ 80%，且 **100% 带页码/章节引用**
5. 端到端单次问答 P95 延迟在可接受范围（内网环境 < 8s，视模型规模调整）
